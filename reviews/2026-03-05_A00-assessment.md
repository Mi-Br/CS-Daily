## Issues

### 🔴 Bug: Slice Aliasing on `n >= len(nums)`

```go
if n > len(nums) {
    return nums  // ← returns the original slice, not a copy
}
```

The caller now holds a reference to the *same underlying array*. If they mutate the result, they corrupt the input — or vice versa. This is a classic Go footgun.

**Fix:**
```go
if n >= len(nums) {
    out := make([]int, len(nums))
    copy(out, nums)
    return out
}
```

Note: changed `>` to `>=` — when `n == len(nums)`, the rest of the function would return all elements anyway, so no need to run the loop.

### 🟡 Algorithm: O(n·k·log k) — there's a better way

Current approach: for each element, append it, sort the whole `out` slice, then truncate. That's O(k log k) per element — total O(n·k·log k).

The canonical solution for "top N" is a **min-heap of size k**:
- Push each element
- If heap size > k, pop the minimum
- Result: heap always holds the k largest seen so far
- Complexity: O(n·log k) — meaningful difference at scale

Go's `container/heap` package covers this. Worth knowing about for A-06+.

For an A-00 assessment this approach is fine — the logic is correct. Just be aware of the tradeoff.

### 🟡 Tests: Length-only assertions don't prove correctness

```go
if len(got) != tc.wantLen {
    t.Errorf("got len %d, want %d", len(got), tc.wantLen)
}
```

`TopN([]int{30, 40, 50}, 2)` returning `[30, 40]` would pass this test. That's wrong output, but the test wouldn't catch it.

Every test case should assert the *actual values*, not just the length. The `TestCorrectOutput` test does this right — apply that pattern everywhere.
